AutoFactory pipeline — Cursor cloud automation

The same Phase 1 chain (research → flag → metrics → tests → review), run as a Cursor cloud Automation instead of a GitHub Action. A PR-opened event fires a fresh cloud sandbox; Cursor's own agent runs the chain, fetching each agent's instructions from LaunchDarkly and using LaunchDarkly MCP + native tools. It produces the same release manifest, so Phase 2 (Beacon) is unchanged. Hover a node for the mechanics.

Phase 1: PR → release resources (Cursor cloud agent)

A Cursor Automation on the "Pull request opened" trigger. Five agents, instructions resolved from LaunchDarkly AI Configs at run time, sequenced by a committed Cursor rule.

Step 1
Developer opens a PR
unflagged feature change
The PR contains the feature change with no flag wiring. Classification, flag creation, code wiring, metric instrumentation, and tests are produced by the agent and delivered as a separate PR.
→
Step 2
Cursor Automation fires
PR opened → cloud sandbox
A Cursor Automation triggered on Pull request opened spins up a fresh cloud sandbox and clones the repo.
  • The committed .cursor/ artifacts (the rule + command) ride along in the clone.
  • .cursor/environment.json runs the install command so the test suite can run later.
  • Loop guard: if the change set already contains a .release-flags/ manifest, the run stops — the automation's own PR would otherwise re-fire this trigger.
→
Step 3
Agent reads the rule
sequencing + tool translation
Cursor's native agent loads .cursor/rules/autofactory.mdc, which owns the chain order, the LaunchDarkly conventions, and a tool-translation table.
  • Prompts stay config: each phase's instructions are fetched from LaunchDarkly via the get-ai-config MCP tool at run time.
  • Tool translation maps the instructions' tool names to native + MCP equivalents (e.g. create_flag → MCP create-feature-flag).
  • Runs in one continuous context — no graph traversal; the rule's prose is the routing.
→
Step 4
Research & Planner
read-only analysis
Config autofactory-research-planner. Native file reads, grep, git diff; no writes.
  • Classifies pr_type, risk_level, change_scope, user-facing impact, and the repo's existing flag/test/telemetry patterns.
  • Short-circuit: if the change is config-only, docs, a dependency bump, infra, or test-only, the agent stops here, comments "no flag needed," and opens no PR.
→
No flag
Stop & comment
no PR opened
When Research concludes the change needs no flag, the chain ends. The agent posts a short comment on the triggering PR saying no flag was needed and does not open a PR. Downstream agents never run.
→
Step 5
Flag Implementer
creates the flag, wires the code
Config autofactory-flag-implementer.
  • Creates a boolean flag via the MCP create-feature-flag tool in the app project, targeting off in all environments.
  • Wraps the new behavior in flag evaluation; the control path (flag off) keeps prior behavior.
  • Idempotent: an existing flag (409) is reused.
→
Step 6
Metrics Author
creates guarded-release metrics
Config autofactory-metrics-author. Metrics are created via the LaunchDarkly MCP metric tool — no REST script.
  • Categories: error (occurrence, lower is better), latency (numeric ms, lower is better), business (occurrence, higher is better).
  • Adds track() calls on the flagged path; each metric's eventKey equals the tracked event name. Key convention <flag>-<category>.
  • Writes the release manifest .release-flags/<id>.json with metricKeys + randomizationUnit, which Phase 2 reads.
→
Step 7
Flag Testing
flag-on and flag-off tests
Config autofactory-flag-testing.
  • Writes paired tests per flagged path: flag on asserts the new behavior, flag off asserts the prior behavior.
  • Uses the repo's existing test framework and runs the suite in the sandbox terminal, fixing failures.
→
Step 8
Code Reviewer
APPROVE / REJECT + risk_level
Config autofactory-code-reviewer. Read-only review of the full change set.
  • Checks logic, error handling, control-path preservation, test quality, and that track() event names match each metric's event_key.
  • Produces review_approved + risk_level. Advisory here — the PR itself is the human gate.
→
Step 9
Cursor opens a PR
new branch + verdict comment
The agent commits its work to its own cursor/… branch and Cursor opens a new PR (not the triggering one).
  • It comments on the triggering PR with the flag key, metric keys, manifest path, and the review verdict (LaunchDarkly links included).
  • Open question: whether an automation can instead push to the triggering PR's branch or stack onto it — today it opens a parallel PR.
→
Step 10
Human merges
flag remains off
A human reviews and merges the AutoFactory PR. The flag is off in every environment, so the deploy serves the control path. The release manifest is now on the default branch where Phase 2 discovery can see it.

Phase 2: deploy → guarded release (Beacon)

Identical regardless of which Phase 1 front end produced the manifest. HTTP service hosted separately; receives deploy notifications, diffs release manifests, triggers and monitors releases via the LaunchDarkly API.

Step 11
CD deploys the services
existing deploy pipeline
The merge deploys through the team's existing CD (Railway in the demo, via its GitHub integration). AutoFactory does not run the deploy; it only consumes a notification that a deploy succeeded.
→
Step 12
Deploy webhook → Beacon
POST {service, sha} + secret
Generic endpoint /flag-releases accepts {service, sha, previousSha?, environment?}, authenticated by a shared secret in a header or query parameter.
  • Provider adapters translate native payloads to the same handler. The Railway adapter reads details.status, details.commitHash, resource.service.name.
  • Only success statuses proceed. Other deploy statuses return 200 with ignored.
  • Providers retry non-2xx deliveries, so all handling is idempotent.
→
Step 13
SHA diff → discovery
previousSha resolution + manifest diff
.release-flags/ is listed at the deployed SHA and at the previous SHA (GitHub contents API). Manifests present only at the new SHA are released.
  • previousSha resolution order: explicit field in the notification, then Beacon's deploy-state store (last SHA per service@environment), else none. With none, all current manifests are treated as new (first deploy).
  • The store keeps last and prior SHAs. Re-delivery of the current SHA re-diffs prior..sha instead of the empty sha..sha.
  • No dependency on the CD system reporting its own deploy history.
→
Step 14
Scope & fullstack check
frontend / backend / fullstack
Each manifest carries a scope; each service registers a side (config/services.yaml).
  • A notification from the wrong side skips the flag; the right side's notification releases it.
  • Fullstack: Beacon reads the other side's status endpoint for its deployed SHA and checks that SHA contains the same manifest file. If absent, the flag waits and is re-evaluated when the other service's notification arrives.
→
Step 15
Release trigger
method and stages by precedence
  • Method precedence: manifest releaseOverrides.releaseMethod, then the flag's configured release policy, else guarded when metricKeys exist, progressive otherwise.
  • Stage precedence: overrides, then policy, then defaults (guarded 20%/50%, progressive 20/50/100%).
  • Guarded stage allocations are limited to 50% by the API; the release serves 100% after the final stage completes.
  • Flags are off at this point, so the semantic patch combines turnFlagOn and startAutomatedRelease in one request.
  • A flag with an in-progress release reports already_running instead of re-triggering.
→
Step 16
Guarded release runs
staged rollout, metric comparison
LaunchDarkly shifts traffic per stage while the Phase 1 metrics compare treatment against control on live traffic.
  • autoRollback is set per metric at trigger time.
  • The randomization unit from the manifest keeps measurement aligned with the rollout.
→
Step 17
Beacon monitors
poll to terminal state
Beacon resolves the release id (per-flag automated-releases list, filtered to in_progress) and polls until a terminal state:
  • completed: rolled out to 100%.
  • reverted: a guardrail metric regressed; traffic returned to control.
  • monitoring_stopped: a human intervened.
  • Polling runs detached from the webhook request. A monitoring failure does not affect the release, which runs server-side.
→
On regression
Auto-rollback
flag reverts to control
A killswitch metric regression reverts the release: traffic returns to the control variation by flag change, with no deploy. Beacon records the reverted terminal state.
