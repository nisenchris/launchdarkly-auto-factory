1. Guiding principles

Three goals drive the design: easy setup, clear customization points, and modularity. The agent steps, scope definitions, and the .release-flags/ pattern are all expected to change, so they're treated as swappable parts behind stable seams, optimized for a design partner installing the prototype with as little friction as possible:

  • Easy setup is a feature, not docs. Pipeline setup is baked in: a bootstrap step (CLI or template repo + GitHub Action) wires up CI, secrets shape, the LD project, and a working demo with one command — sane defaults, escape hatches, “simple but not simpler.”
  • Customization points are explicit and few. Every knob a partner is expected to turn lives in a small, documented config/ surface — not buried in code. Agent instructions, scopes, and the release-config source are all config; the approval mode is a LaunchDarkly flag.
  • Modular: adapters at every external boundary. CI (GitHub Actions), CD (Railway), the LD API (agent execution + flags/metrics), and the release-config source sit behind interfaces, so a partner can swap one without touching orchestration logic. (The LLM runtime itself is LD-hosted, so it's not our boundary to adapt.)
  • Agent steps are a registry, not a hard-coded pipeline. Adding, splitting, or reordering agents is a config change, not a refactor.
  • Public-by-default, private-by-exception. The repo is public and the source material is the internal build, so the boundary between shareable prototype and proprietary source is structural, not a matter of remembering.
  • Idempotent & reversible. CI re-runs must not leave orphaned flags or duplicated resources behind.

2. Architecture at a glance

Agent runtime: Vega (LaunchDarkly-hosted)
AgentControl configs and Agent Graphs are configuration, not a runtime (the docs: “LaunchDarkly does not execute graphs or manage state”). That execution substrate is Vega — LaunchDarkly's AI, which runs Claude on LD's Bedrock instance. Throughout this doc, “call Vega” means a request to a LaunchDarkly API endpoint that triggers the agent work. The prototype's job in Phase 1 is therefore integration glue, not a runner: a GitHub Action assembles PR context (diff, files, metadata) and calls Vega, which runs the agent graph and returns the agents' output (proposed changes + a risk score + an APPROVE/REJECT). No agent loop ships in this repo.
What changed in the build (2026-06-09)
This section describes the original Vega-only plan. The build pivoted to a provider seam (AgentRunner): the default is a local Anthropic tool-use loop with capability-gated sandbox tools, and Vega is the alternative — selected at runtime by the auto-factory-ai-provider LaunchDarkly flag. So an agent loop does ship in this repo now (packages/shared/src/anthropic/); the LD AI SDK still resolves the agent configs/graphs and records metrics natively. See ADR 0005.
Config provisioning: the bridge service
The agent configs must land in the LD environment the running prototype draws from. A small bridge service handles this over LaunchDarkly's MCP / API, and is non-agentic: it keeps local canonical copies of the recommended starting configs + graphs and provisions them. It's parameterized on source and target, both configurable to any LD instance/environment:
  • LD team's source of truth: LaunchDarkly's internal instance (the dev's existing work) — a different, non-production API base URL.
  • Customers' source of truth: the LD demo account (production API).
Two distinct touchpoints: provisioning configs into an env (bridge, setup-time) and executing them from that env (LD-hosted, run-time).

LaunchDarkly supplies: AgentControl configs (per-agent model, instructions, tools), the Agent Graph (nodes = agents, edges = handoffs, a root, mutable context keyed by config), and Vega — the hosted execution of that graph, reached via a LaunchDarkly API endpoint. The prototype supplies the GitHub Action that feeds Vega context, the bridge that provisions configs, and the Phase 2 release orchestrator (Beacon).

4. External sources & the public/private boundary

The prototype draws from repos, updatable reference pages, and LD configs — some of it proprietary, while this repo is public. The boundary is structural:

MaterialWhere it livesHow it syncs
Public repos we reference sources/repos/ Vendored manually, pinned in sources/manifest.yaml (synced on demand, not at build time)
Reference pages that change (docs, confluence) Public → sources/; proprietary → reference-private/ Synced snapshots, never live-fetched at build time (reproducibility)
LD configs (flags, AI configs, graphs) Live in the source LD project; not committed (the runtime path uses bridge seed) Pulled via the config bridge (sync / seed) over the LD REST API; the source LD project is the sanitization boundary
Internal build, orchestrator spec, Spinnaker details reference-private/ only Manual; inspiration, not rigid patterns — informs design, never the source of public code/comments
Secrets (LD tokens, Bedrock/AWS creds) Env / CI secrets only .env.example documents shape; real values never in repo

Sources are pinned snapshots, refreshed manually only on word of a critical change — not live-fetched — so the prototype stays reproducible.

Public-repo risk that isn't obvious
The leak surface isn't just files — it's also agent prompts and AgentControl instructions, which can encode proprietary release logic, and code comments / commit messages that paraphrase internal docs. A scripts/check-public pre-commit/CI guard fails if anything resembling internal identifiers, the real internal tool names, or reference-private/ content appears in tracked files. Worth having early, because retroactively scrubbing git history of a public repo is painful.

5. Phase 1 — automatic resource creation

The agent flow maps onto the Agent Graph that Vega executes: each agent is an AgentControl agent-mode config (instructions + tools), edges carry handoffs, and Vega threads the mutable context. Flag creation and metric creation are separate agents, giving five sequential nodes:

  1. Research & Planning — explores the codebase, decides whether the PR needs a flag at all, computes a risk score, and produces the implementation brief. This is the triage step: if it concludes no flag is needed, the graph stops here and the downstream agents never run. Triage lives here because deciding flag-worthiness requires understanding the code change — the same work Research already does — so a cheaper pre-filter would only risk false negatives.
  2. Flagging — creates the flag in LD and wires the PR code to evaluate it.
  3. Metrics — creates the metrics that measure the flag and wires up instrumentation. Runs after Flagging and consumes the created flag key via the graph handoff.
  4. Testing — general coverage plus per-variation flag-path tests.
  5. Code Review — independent quality analysis → APPROVE / REJECT.

Idempotency: CI re-runs on new commits must upsert, not duplicate. Flag keys are derived from stable inputs so a second run is a no-op rather than a second flag.

Approval modes — driven by a LaunchDarkly flag

The active approval mode lives in a LaunchDarkly flag (dogfooding the product), scoped per-repo, with a hardcoded-config fallback. The Research agent's risk score is the input the Middle mode gates on. The default is Yolo.

ModeBehaviorNotes
YoloAuto-apply everything Review approvesDefault. Flag variation served to everyone unless overridden
MiddleGate on the Research agent's risk score: low auto-applies, high needs a humanThresholds TBD (deferred — default is Yolo)
ManualHuman approves every approved changeApproval happens in GitHub

6. Phase 2 — automatic releases

The internal system runs on Spinnaker; the prototype maps the same concepts onto Railway. The orchestrator is named Beacon.

  • Notifier → a post-deploy step in Railway that POSTs the deployed SHA range to Beacon.
  • Beacon → a small service: receives notifications, discovers new release-flags, routes by scope, triggers releases.
  • Fullstack check → a stateless cross-service SHA comparison, re-derived on each notification.

Railway fits a lightweight, low-friction prototype (push-to-deploy, simple post-deploy hooks, minimal partner setup) and aligns with the easy-setup goal. Both demo services deploy via Railway: Phase 2's coordination only needs two independently-deployable services that each emit their own post-deploy notification, which two Railway services provide. (Production frontends often ship to a CDN/edge host instead, but that doesn't change the SHA-range notification or the fullstack scope logic.) The CD adapter keeps us portable regardless.

7. Designing for design partners

The prototype ships to early design partners, with the LD team hands-on but wanting minimal friction. That makes three things acceptance criteria, not nice-to-haves.

GoalWhat it means concretelyHow the design delivers it
Easy setup A partner goes from clone to a working flow with as few manual steps as possible bootstrap/: one command wires the CI workflow, LD env, secrets shape, and a runnable demo; preflight checks fail loudly with fixes
Clear customization The knobs partners are meant to turn are obvious and isolated A small documented config/ surface (agent instructions, scopes, release-config source) plus the approval-mode LaunchDarkly flag — code stays untouched for normal customization
Modular Swap CI/CD/release-source without a rewrite Adapters at every external boundary; agent steps as a registry
"Simple but not simpler"
Baking in setup automation can hide exactly the seams a design partner needs to adapt to their own stack. The resolution is defaults that are one layer deep — the bootstrap generates real, legible config files the partner can then edit, rather than magic that only works on the golden path.

8. Risks & design considerations

Things the build needs to handle deliberately. None block scaffolding, but each is a real design choice rather than an implementation detail.

Code delivery must not loop CI

Wiring flags into the PR means writing code onto a human's branch. A bot commit re-triggers CI (risk of an infinite loop), can conflict with in-flight work, and changes the diff under the developer. Whatever the delivery mechanism (bot commit, GitHub suggestions, or a stacked PR — still open, see §9), it needs a CI-loop guard. Suggestions or a stacked PR keep it non-destructive.

Release-flag identity should be robust

Keying release config by PR number (pr-N.json) is fragile: PR numbers are repo-local and meaningless after merge, one deploy can contain many PRs, and squash/rebase/force-push rewrite the SHAs discovery diffs against. Keying by flag key (or a content hash) is more robust. The release-config source is an adapter so the file-in-repo pattern can be swapped for an LD-native source later without reworking Beacon.

Fullstack coordination needs a backstop

Re-deriving readiness from currently-deployed SHAs on each notification is a clean event-driven design, but the “wait for the other pipeline's Notifier to re-evaluate” path means a lost notification could silently strand a release. The coordination logic needs a retry/timeout backstop or an explicit at-least-once delivery guarantee.

Why flagging & metrics are split
They're genuinely different jobs: flag creation is control-flow wiring and variation shape; metric creation is what to measure, event instrumentation, and tying metrics to the flag's release. Bundling them tends to produce a brief strong on one and thin on the other. Separate agents can be tuned, evaluated, and swapped independently — the modularity goal. The cost is one handoff: the Metrics agent needs the flag key the Flagging agent created.

Open questions

  1. How does the Flagging agent deliver code changes to the PR? Bot commit, GitHub suggestions, or a stacked PR — pending the source material (a teammate used GitHub's Agent HQ, a candidate not a commitment). Whatever we pick needs a CI-loop guard (§8).
  2. Risk-score thresholds & factors for Middle mode. Deferred — default is Yolo. Eventually: what factors feed the score, and what threshold flips auto-apply → human review?
  3. Agent-graph CRUD over the API. A build-time detail: confirm the bridge can create/update Agent Graphs programmatically. The MCP surfaces AI-config CRUD, but graph CRUD may be raw-API. Graphs are stored locally regardless, so worst case we push them via REST.
